# Fast-path canonical device-slot mapping tensors

Status: done
Type: AFK

## What to build

Reduce steady-state host overhead in device-slot KV writes by directly using
vLLM's CUDA `slot_mapping` tensor when it is already the canonical shape and
type expected by the KCMM device-slot write ABI.

## Why

After Issue 38, the latest host-profile run shows the next small write-side
device-slot preparation cost:

- `write_device_slot_prepare_tensor=1.939ms` total over `384` calls
- `write_prepare_rows=2.708ms` total over `384` calls

The current device-slot path always calls `slot_mapping.detach().reshape(-1)`
before checking dtype and contiguity. If vLLM 0.6.1 already passes a 1-D,
CUDA-resident, contiguous `torch.int64` tensor, KCMM can use that tensor
directly and avoid per-write tensor wrapper work.

## Acceptance criteria

- [x] Use the original `slot_mapping` tensor directly when it is CUDA, 1-D,
  contiguous, and `torch.int64`.
- [x] Preserve fallback behavior for non-int64, non-contiguous, or non-1-D slot
  mappings.
- [x] Report direct, reshape, dtype-conversion, and contiguous-copy counts in
  the write tracker report and A/B contract summary.
- [x] Validate short vLLM smoke, performance-clean, performance-clean stress,
  and host-profile gates.

## Boundaries

- This does not change the device-slot write ABI.
- This does not change `slot_mapping` values or padding handling.
- This does not relax CUDA device validation between `slot_mapping` and K/V
  tensors.

## Implementation

- Added `_prepare_device_slot_tensor()` to directly reuse canonical
  CUDA/1-D/contiguous/`torch.int64` `slot_mapping` tensors.
- Preserved fallback behavior for non-canonical tensors via `reshape(-1)`,
  dtype conversion, and contiguous copy.
- Added write tracker counters for direct, reshape, dtype-conversion, and
  contiguous-copy prepare paths.
- Surfaced those counters through the A/B contract summary and
  performance-clean requirements.
- Made the performance-clean gate fail if current vLLM 0.6.1 device-slot writes
  do not all use the direct prepare fast path.

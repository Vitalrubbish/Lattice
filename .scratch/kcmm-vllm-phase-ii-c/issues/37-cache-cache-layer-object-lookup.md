# Reject cache-layer object lookup

Status: done
Type: AFK

## What to investigate

Evaluate whether KCMM read/write cache-layer lookup can use Python object
identity as a fast path before falling back to data-pointer based layer
registration.

## Why

After Issue 36, the latest host-profile run still shows small but steady
request-path overhead in layer lookup:

- `read_layer_for_cache` around `1.0ms` total
- `write_layer_for_cache` around `1.0ms` total

The trackers already cache layer ids by `(key_cache.data_ptr(),
value_cache.data_ptr())`, but every read/write seam still calls `data_ptr()` on
the vLLM KV cache tensors to reach that cache. If the vLLM cache tensor Python
objects were stable, an object-id fast path could avoid the per-seam pointer
calls after the first lookup for each layer.

## Result

- [x] Implemented a local object-id/weakref prototype for read and write layer
  lookup.
- [x] Validated with short vLLM smoke.
- [x] Rejected the optimization because the object-id fast path had zero hits.
- [x] Removed the prototype before commit.

## Evidence

Short vLLM smoke with the prototype showed:

- Read cache-layer pointer cache entries: `2`
- Read object-cache hits/misses: `0/6`
- Write cache-layer pointer cache entries: `2`
- Write object-cache hits/misses: `0/8`

The pointer cache confirms that the underlying KV cache data pointers are
stable. The object-cache misses show that vLLM/PyTorch passes fresh Python
tensor wrapper objects at these seams, so object identity is not a useful cache
key.

## Boundaries

- This does not change the KCMM C/Rust ABI.
- This does not change layer numbering or KV storage layout.
- This intentionally leaves the existing data-pointer cache unchanged.

## Follow-up

Do not pursue Python object-identity caching for vLLM KV cache tensors in this
0.6.1 eager integration. Future cache-layer lookup work should target avoiding
or combining data-pointer calls through a different seam, not object identity.

# Fast-path canonical KV write row tensors

Status: done
Type: AFK

## What to build

Reduce steady-state host overhead in KCMM KV writes by directly viewing vLLM's
canonical contiguous key/value tensors as row-major batches, instead of calling
`contiguous()` on every write seam.

## Why

After Issue 39, the latest host-profile run shows `write_prepare_rows` as the
next small write-side tensor preparation cost:

- `write_prepare_rows=2.705ms` total over `384` calls
- `write_device_slot_prepare_tensor=0.790ms` after the slot mapping fast path

vLLM 0.6.1 currently passes CUDA-resident, contiguous FP16 key/value tensors
whose first dimension matches `slot_mapping` batch. In that canonical case KCMM
can use `view(batch, -1)` directly and skip the per-call `contiguous()` wrapper
work.

## Acceptance criteria

- [x] Use a direct `view(batch, -1)` path when key/value tensors are contiguous
  and batch-aligned.
- [x] Preserve fallback behavior for non-contiguous or non-canonical tensors.
- [x] Report direct, fallback, and contiguous-copy row prepare counters in the
  write tracker report and A/B contract.
- [x] Make the single-request performance-clean gate require the direct row
  prepare fast path to be used and dominate fallback calls.
- [x] Validate short vLLM smoke, performance-clean, performance-clean stress,
  and host-profile gates.

## Boundaries

- This does not change the KCMM write ABI.
- This does not change row contents, dtype validation, stream selection, or
  device-slot table semantics.
- Current vLLM prefill writes can pass non-contiguous batch>1 key/value tensors
  with gaps between rows; those must keep using the contiguous-copy fallback
  because the current KCMM write ABI assumes compact source rows.
- Concurrent stress scheduling can keep all writes in non-contiguous batch>1
  form, so the stress gate validates row prepare accounting rather than
  requiring direct-row dominance.
- Correctness paths still fall back to contiguous copies if vLLM changes tensor
  layout.

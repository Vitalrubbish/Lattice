# Cache device-slot table device key

Status: done
Type: AFK

## What to build

Reduce steady-state host overhead in device-slot KV write table lookup by
tracking cached table device identity with the CUDA device index instead of
recomputing and comparing `torch.device` strings on every write seam.

## Why

After Issue 36, the latest host-profile run still shows write-side table lookup
cost:

- `write_device_slot_table_lookup=4.658ms` total

The table cache is already keyed by block-state epoch and device, but cache hits
compare device identity by converting the current and cached tensor devices to
strings. The write path already validates `slot_mapping` and K/V tensors are on
the same CUDA device and has that integer device index available.

## Acceptance criteria

- [x] Store the cached device-slot table device as an integer CUDA device index.
- [x] Use the validated device index for cache-hit device comparison.
- [x] Preserve block-state epoch invalidation and table rebuild behavior.
- [x] Report the cached table device index in the write tracker report.
- [x] Validate short vLLM smoke, performance-clean, performance-clean stress,
  and host-profile gates.

## Boundaries

- This does not change the device-slot write ABI.
- This does not remove `kcmm_block_state_epoch()` validation.
- This does not change device-slot offset/valid table contents.

## Implementation

- Added `_last_device_slot_table_device_index` to the write mirror tracker.
- Changed device-slot table cache-hit checks to compare the validated CUDA
  device index instead of converting cached tensor devices to strings.
- Added `device_slot_table_device_index` to the write tracker report, A/B
  contract summary, and performance-clean requirements.
- Added a performance-clean failure if the device-slot table device index is
  missing or negative.
